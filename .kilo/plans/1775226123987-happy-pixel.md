# Client-Sided Operations that should be server-authoritative

# Analysis Document

## Executive Summary

This document identifies client-sided operations in the `rose-offline-client` that should be handled on the server for security and game integrity.

## Critical Issues (High Priority)
### 1. Passive_recovery_system.rs (HP/MP Regen)
- **File**: `src/systems/passive_recovery_system.rs`
- **Problem**: HP and MP regeneration is fully client-side
- **Client**: `health_points.hp += recovery` (lines 28-31)
- **Client**: `mana_points.mp += recovery` (lines 34-40)
- **Server**: `game/systems/passive_recovery_system.rs` has server-side equivalent
- **Impact**: Client can manipulate HP/MP values, heal to any with infinite health/mana. Speed hacking possible (auto-regeneration exploits)

- **Risk**: HIGH

### 2. status_effect_system.rs (Status effects)
- **File**: `src/systems/status_effect_system.rs`
- **Problem**: HP/MP modifications from status effects (IncreaseHp, IncreaseMp, poison damage) done client-side
- **Client**: `status_effect.apply_status_effect()` modifies HP/MP (lines 89-93)
- **Server**: `game/systems/status_effect_system.rs` has server-side equivalent
- **Impact**: Client can apply fake buffs, manipulate regeneration rates, poison immunity without dying
- **Risk**: High

### 3. monster_separation_system.rs
- **File**: `src/systems/monster_separation_system.rs`
- **Problem**: Monster collision separation modifies entity positions client-side
- **Client**: `position.x += separation` (line 50)
- **Server**: None
- **Impact**: Client can push monsters around,, creating position exploits
- **Risk**: Medium

### 4. quest_trigger_system.rs (Quest Rewards)
- **File**: `src/systems/quest_trigger_system.rs`
- **Problem**: Quest rewards are applied client-side without server validation
- **Client**: `quest_apply_rewards()` (lines 69-101) applies XP, money, items, skill points directly
- **Server**: `game/systems/quest_system.rs` handles this server-side
- **Impact**: Client can grant themselves arbitrary rewards (items, XP, skill points, money)
- **Risk**: High

### 5. flight_movement_system.rs (Flight Physics)
- **File**: `src/systems/flight_movement_system.rs`
- **Problem**: Entire flight system is client-authoritative
- **Client**: All physics (acceleration, speed, position) done locally
- **Server**: `game/systems/update_position_system.rs` only handles X/Y ( no Z-axis)
- **Impact**: Client can fly anywhere at any speed with no server validation
- **Risk**: High

### 6. flight_toggle_system.rs (Flight Toggle)
- **File**: `src/systems/flight_toggle_system.rs`
- **Problem**: Server never knows when player enters/exits flight mode
- **Client**: `FlightState.is_flying` togg locally
- **Server**: None
- **Impact**: Player can toggle invulnerability and fly through walls
- **Risk**: High

## Detailed Analysis: Client-Side vs Server-Side

### Combat System
**Client:**
- Receives `DamageEntity` from server with pre-calculated damage
- Applies damage to local HP
- Spawns visual effects

**Server:**
- Calculates damage using `AbilityValueCalculator`
- Validates target (team, entity type, alive)
- Applies damage to entity HP
- Sends `DamageEntity` to clients

**Assessment**: Damage calculation is server-authoritative. Client only visualizes. ✅ Good

### Movement System
**Client:**
- Sends move commands to server
- Interpolates position locally for smooth movement
- Handles collision locally

**Server:**
- Calculates movement based on server's `MoveSpeed`
- Does NOT validate position bounds
- Does not validate distance to target
- `MoveCollision` accepts client position directly

**Assessment**: Server movement is authoritative, but position validation is weak. Movement collision is server-synced. ⚠️ Needs bounds checking

### Items/Inventory
**Client:**
- Sends item use/equip commands to server
- Displays inventory UI

**Server:**
- Validates item existence in inventory
- Validates item types for equipment slots
- Validates job/union/stat requirements
- Updates inventory

**Assessment**: Item system is server-authoritative. ✅ Good

## Recommendations

### Immediate (Critical)
1. **Passive Recovery**: Move HP/MP regeneration to server-side only
2. **Status Effects**: Move HP/MP modifications from status effects to server-side only
3. **Quest Rewards**: Add server RPC for for quest rewards
4. **Flight System**: Add server sync for flight state and position
5. **Monster Separation**: Remove client-side or handle server-side

6. **Cooldowns**: Add server-side cooldown tracking

### Near-term (High Priority)
7. **Ability Values**: Sync client stats with server more frequently
8. **Position Bounds**: Add validation for x,y coordinates
9. **Distance Checks**: Add distance validation for attacks/interactions
10. **Collision Position**: Add server validation for `MoveCollision` position
11. **Rate Limiting**: Add rate limiting for chat/commands

### Long-term (Architecture)
12. Consider client-side prediction with server reconciliation for position/movement
13. Periodic server sync of HP/MP/status effects to client
14. Consider adding encryption/signing to protocol
